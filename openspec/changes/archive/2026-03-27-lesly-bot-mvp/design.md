## Context

Greenfield проект — Telegram бот для соревновательного трекинга подходов к девушкам. Пользователи добавляют записи через slash-команды, получают очки, соревнуются на лидерборде в Mini App. Система должна быть расширяемой: новые типы сущностей добавляются через INSERT в БД без изменения кода.

Текущее состояние: пустой репозиторий, нет существующей кодовой базы.

```
┌─────────────┐     ┌──────────────────┐     ┌───────────┐
│  Telegram    │────▶│  aiogram 3 bot   │────▶│ Supabase  │
│  Users       │◀────│  + aiohttp API   │◀────│ (Postgres)│
└─────────────┘     └──────┬───────────┘     └───────────┘
                           │ serves
                    ┌──────▼───────┐
                    │  React Mini  │
                    │  App (Mantine)│
                    └──────────────┘
```

## Goals / Non-Goals

**Goals:**
- Полностью рабочий бот с динамическими сущностями из БД
- Mini App с лидербордами, просмотром и отправкой контактов
- Docker-based деплой одной командой
- Расширяемость: новая сущность = INSERT в entity_types

**Non-Goals:**
- Админ-панель для управления entity_types (используется Supabase UI)
- Авторизация/регистрация пользователей (авто-регистрация при /start)
- Push-уведомления и реал-тайм обновления
- Мультиязычность (только русский)
- Rate limiting и защита от спама

## Decisions

### 1. Единый процесс: aiogram + aiohttp
**Решение**: Bot и API сервер живут в одном Python процессе.
**Альтернатива**: Отдельные сервисы (бот + API).
**Обоснование**: Для MVP избыточно разделять. Один процесс упрощает деплой и позволяет API напрямую использовать bot instance для отправки сообщений (send-contacts). aiohttp запускается как фоновая задача при старте бота.

### 2. Единая таблица entries вместо таблицы на сущность
**Решение**: Все записи (girl, beautiful_girl, будущие) хранятся в одной таблице `entries` с FK на `entity_types`.
**Альтернатива**: Отдельная таблица для каждого типа (girls, beautiful_girls, dates...).
**Обоснование**: Единая таблица масштабируется без миграций при добавлении новых сущностей. Запросы агрегации проще через JOIN. Если потребуются специфичные поля для типа — можно добавить JSONB колонку `extra_data`.

### 3. Динамическая загрузка entity_types при старте + кэш
**Решение**: При старте бота загружаются все entity_types из БД. Кэшируются в памяти. Команды регистрируются динамически. Перезагрузка кэша по команде или рестарту.
**Альтернатива**: Чтение из БД при каждой команде.
**Обоснование**: entity_types меняются редко (ручной INSERT в Supabase). Кэш убирает лишние запросы. При добавлении новой сущности нужен рестарт бота (приемлемо для MVP).

### 4. Валидация initData на сервере
**Решение**: Все API эндпоинты валидируют Telegram initData (HMAC-SHA256) в aiohttp middleware.
**Альтернатива**: Без валидации (доверять клиенту).
**Обоснование**: Безопасность — без валидации любой может дёргать API. Стандартный подход для Telegram Mini Apps.

### 5. React + Mantine + Recharts для Mini App
**Решение**: Mantine v7 как UI-кит, Recharts для гистограмм, Vite для сборки.
**Альтернатива**: Ant Design, shadcn/ui, Chart.js.
**Обоснование**: Mantine — современный, лёгкий, хорошая документация. Recharts — декларативный, React-нативный. Vite — быстрая сборка.

### 6. Статика Mini App раздаётся через aiohttp
**Решение**: Собранная React-аппка (dist/) раздаётся как статика через aiohttp static routes.
**Альтернатива**: Отдельный nginx контейнер.
**Обоснование**: Один контейнер, простой деплой. Для MVP нагрузка минимальна. При росте можно вынести в nginx.

### 7. asyncpg для работы с БД
**Решение**: Прямое подключение к PostgreSQL через asyncpg (connection pool).
**Альтернатива**: supabase-py SDK, SQLAlchemy async.
**Обоснование**: asyncpg — максимальная производительность и контроль. supabase-py добавляет лишнюю абстракцию для прямых SQL-запросов. ORM избыточен для MVP с 4 таблицами.

## Risks / Trade-offs

**[Рестарт при новой сущности]** → Приемлемо для MVP. В будущем можно добавить hot-reload через admin-команду.

**[Единый процесс — single point of failure]** → Docker restart policy `unless-stopped`. Для MVP достаточно. При росте можно разделить на сервисы.

**[Нет rate limiting]** → Любой пользователь может спамить команды. Митигация: добавить rate limiter в будущем через aiogram middleware.

**[asyncpg без ORM]** → Ручные SQL-запросы. Для 4 таблиц это проще, но при росте схемы может стать проблемой. Митигация: сервисный слой абстрагирует SQL.

**[initData expiration]** → Telegram initData имеет ограниченный срок жизни. Нужно обрабатывать expired initData gracefully.

## Migration Plan

Greenfield — нет миграции. Порядок деплоя:
1. Создать Supabase проект, выполнить миграцию `001_initial.sql`
2. Заполнить `.env` с BOT_TOKEN, DATABASE_URL, WEBAPP_URL
3. `docker-compose up -d`
4. Настроить Mini App URL в BotFather

Rollback: `docker-compose down`, DROP таблиц в Supabase.
